# F.21A — מסך מורה בשפת ראמ"ה
## Wireframe + UX Spec · 27.5.2026 · שלב א' (לפני קוד)

> **מה זה המסמך הזה:** spec עיצובי שמוביל את בניית F.21A. **לא קוד** — markdown + ASCII mockups. סוכן הקוד יקבל את המסמך הזה ויבנה לפיו.
>
> **למה צריך:** היום teacher-live.html מציג את הכיתה דרך עדשת "המשחקונים של אבני יסוד" (5 אותיות באי 3 + per-letter accuracy). זאת **שפה פנימית**. F.21A מציג את אותה כיתה דרך עדשת **משרד החינוך + ראמ"ה** — 10 משימות × 3 פעימות — שהיא השפה שהמורה מדברת בה עם המפקחת, ההורה, והקולגה.
>
> **תלויות שנסגרו:** A.1 (BKT-per-strand) ✓ · A.3 (EPA) ✓ · A.4 (Sub-BKT 22 אותיות) ✓ · A0.3 (Mastery משולש) ✓. כל ה-API מוכן.

## §1 — הקשר: איפה F.21A יושב

| שאלה | תשובה |
|---|---|
| איזה מסך זה מחליף? | סקציה 6 ("אבני יסוד נסגרות — מבדק ראמ"ה") ב-`teacher-live.html`, שהיא כרגע "תצוגת בדיקה למפתחים" |
| איזה מסך זה משאיר? | סקציות 1-5 של `teacher-live.html` (תקציר, פר-אות, פר-משחקון, דגלים, אירועים) — נשארות **כ-view משני** ("מבט פנימי / דיאגנוסטיקה") |
| מי קהל היעד? | מורה כיתה א'. **לא** המפקחת, **לא** ההורה (יש לזה view-ים אחרים שיבנו בנפרד — משימה B.10) |
| מתי המורה משתמשת בו? | בוקר, לפני שיעור. 2 דקות עם קפה. או 1.11/1.2/1.6 — סיום פעימה |
| מה השאלה שהיא שואלת? | "מי דורש תשומת לב היום?" + "מי לא מוכנה לפעימה הבאה?" |

**עקרון מנחה:** המורה מקבלת תשובה לשאלה הזו תוך **5 שניות** מפתיחת המסך.

## §2 — 5 ההחלטות שנסגרו עם מיטל

| # | החלטה | תשובה | נימוק |
|---|---|---|---|
| 1 | מבנה תצוגה ראשי | **טבלה** (תלמידות × משימות, צבע פר תא) | היחיד שעונה תוך 5 שניות גם על "מי לא תקין במשימה X" וגם על "במה תלמידה Y חלשה" · רטרו-פיט עם סקציה 6 הקיימת · שפה מוכרת למורה |
| 2 | View ראשי בכניסה | **שני כפתורים שווי-משקל** בראש המסך: `[מבט כיתה]` + `[מבט תלמידה]` | בלי לכפות ברירת מחדל · המורה בוחרת לפי המשימה שלה |
| 3 | רמת פירוט פר תלמידה | **הכל + Confidence indicators (✅🟡⚫)** | 5 strands + 22 letters + EPA dominant pattern + per-RAMA-task status · עם ביטחון פר כל סטרנד · חוסם False Negatives ב-cold-start |
| 4 | תצוגת פעימה | **שניהם — toggle/tabs להחלפה** | `[מבט יומי]` (מה קרה אתמול/השבוע) + `[Snapshot פעימה]` (BOY/MOY/EOY) · עלות שני tabs בלבד |
| 5 | פרטיות תלמידים | **URL נפרד למורה + סיסמה (PIN) ב-sessionStorage** | פיילוט קטן · ב-2 התקנים שונים · "שער נפילה" מספיק · roadmap לתעדוף ב-Apps Script בשלב ב' · F.21A **לא** לינקד משום מסך תלמידה |

**הערות חשובות לגבי הטבלה:**

- **שמירה רוחבית במובייל:** הטבלה גוללת אופקית. **עמודת שם התלמידה sticky מימין** (RTL). שם תלמידה תמיד גלוי.

- **לחיצה על כותרת משימה** (`מ1`, `מ2`...) → modal/דריל-דאון עם רשימת כל הילדות מסודרות לפי סטטוס במשימה הזו + הצעת אינטרבנציה אם יש דפוס משותף.
- **לחיצה על תא** (ילדה × משימה) → tooltip קצר עם המספרים: "מאיה, משימה 1: 22 אותיות מזוהות מתוך 22 · BKT 92% · שטף 1.8 שנ' פר אות · ✓ עברה רף".

## §4 — זרימות "מה קורה כש..."

### Use Case 1 — מורה נכנסת בבוקר (תרחיש 80%)

1. פותחת `teacher-rama.html` במחשב המורה (bookmark).
2. אם לא הזינה PIN בלשונית הזו — Privacy Gate (§3.4). אחרת — דילוג ישיר.
3. רואה Class View · Daily mode · פעימה 1 פעילה (ברירת מחדל).
4. **סורקת בעין** את עמודת משימה 1 (היחידה שמלאה כרגע) — מאתרת את 3 התאים האדומים (✗).
5. לוחצת על שורת הסיכום: `[פתחי קבוצת תמיכה]`. רואה modal עם 3 השמות + הצעת אינטרבנציה משותפת.

**זמן לתשובה:** 10-15 שניות מפתיחת המסך.

### Use Case 2 — בודקת תלמידה ספציפית

1. ב-Class View, לוחצת על שם תלמידה ("נועה") בעמודה הימנית.
2. או לוחצת על כפתור הtoggle העליון `[מבט תלמידה]` ובוחרת מרשימה.
3. נפתח Student View (§3.2). רואה את 5 הסטרנדים + 22 אותיות + EPA + 10 משימות.
4. בוחנת את EPA — רואה דפוס "medial" — לוחצת `[פתחי סקריפט אינטרבנציה]`.

### Use Case 3 — רואה דגל אוטומטי בכיתה

1. ב-Class View, בכותרת הראשית — `🟡 דגל חדש: 3 ילדות עם דפוס Shape דומה`.
2. לוחצת על הבאנר.
3. נפתח modal עם 3 השמות + הצעת אינטרבנציה "תאומות עם תנועה" + סקריפט.

### Use Case 4 — סיום פעימה (29.10, מורה רוצה לדעת מי "סגרה" פעימה 1)

1. פותחת F.21A.
2. עוברת ל-`[Snapshot פעימה ▣]`. בוחרת פעימה 1.

### ⚠ פונקציה חדשה שדרושה — `checkRamaTaskStatus`

הפונקציה הזו **לא קיימת היום**. יש פונקציה דומה: `AvneiMasteryCheck.checkMastery(studentId, islandId)` שעובדת **פר אי**, אבל F.21A דורש מענה **פר משימת ראמ"ה** (אחת ל-1+ איים).

**חתימה דרושה:**
```
AvneiMasteryCheck.checkRamaTaskStatus(studentId, ramaTaskId)
  → { status: 'pass'|'near'|'fail'|'cold',
      value: number,         // לדוגמה: 22 (אותיות מזוהות)
      threshold: number,     // לדוגמה: 18
      confidence: 'high'|'med'|'low',
      contributingIslands: [3, ...],
      reason: 'BKT 92%, fluency steady, 22/22 letters above threshold' }
```

**מימוש מוצע** (אחריות סוכן הקוד):
- iterate על האיים המתאימים ל-`ramaTaskId` (יש מיפוי ב-`ISLAND_TO_RAMA`).
- aggregate התוצאות (כל האיים pass? → pass. חלקם pass חלקם near? → near. וכו').
- "cold" אם **אף אחד** מהאיים המקבילים לא נמדד (פחות מ-10 ניסיונות בסטרנד הרלוונטי).
- threshold לקוח ישירות מ-spec משימת ראמ"ה (טבלה ב-`2026-05-26-partners-review-v3.md` §3).

**מיקום מוצע ל-implementation:** `js/shared/mastery-check.js` (הרחבה של הקובץ הקיים — לא קובץ חדש).

## §6 — Privacy Implementation Details

### עקרונות

1. **F.21A נגיש רק דרך URL נפרד:** `teacher-rama.html` (לא קישור משום מסך תלמידה).
2. **PIN gate על כניסה ראשונה ללשונית:** modal חוסם עד שמזינים PIN נכון.
3. **PIN נשמר ב-`sessionStorage`** (לא localStorage) — נמחק אוטומטית בסגירת הלשונית.
4. **PIN client-side:** מאומת מול hash hard-coded (`SHA-256` של ה-PIN). זה לא הצפנה אמיתית — זה **שער UX**.
5. **localStorage קריא מכל דף:** זאת מגבלה ידועה. F.21A לא פותר את זה — רק מקטין את ההזדמנות (אין link, יש PIN).

### מה בונים ב-F.21A

```javascript
// בראש teacher-rama.html, לפני render:
const TEACHER_PIN_HASH = '<sha256_of_PIN_default_4521>';  // הסבר: PIN נשלח למורה כשהיא נרשמת לפיילוט
function requirePIN() {
  if (sessionStorage.getItem('teacher_authed') === '1') return true;
  // show modal, get input, hash, compare
  // on success: sessionStorage.setItem('teacher_authed', '1');
}
```

### מה לא בונים (משאירים לשלב ב')

- ✗ Apps Script backend עם token חתום (post-pilot).
- ✗ ניהול חשבונות מורות (post-pilot).
- ✗ refresh tokens / expiration.
- ✗ הצפנת localStorage (post-pilot, אם הרבים-תלמידות-במכשיר-אחד יהפוך לדפוס שכיח).

### Edge cases ב-PIN

| מצב | תגובת המערכת |
|---|---|
| PIN שגוי | "PIN שגוי. נסי שוב." (אין lockout — זה pilot) |
| מורה שכחה PIN | URL נפרד `teacher-rama.html?reset=1` עם הודעה: "פני להנהלה לקבלת PIN חדש" |
| ילדה מקלידה PIN בטעות | Privacy gate מנע — PIN ספציפי, לא ניתן לנחש ב-3 ניסיונות |
| מורה רוצה להראות ל-2 הורים בו-זמנית | אותו PIN, בכל לשונית — sessionStorage נפרד |

## §7 — Confidence Indicators — Behavior Spec

הסימן ליד שם תלמידה (Class View) ובכל סטרנד (Student View) קובע **כמה אמינה הקריאה**.

### חוקי הסיווג (אחריות API)

```
function getConfidence(attemptCount, lastActivityDays):
  if attemptCount < 10:
    return '⚫' (טרם נמדד)
  if attemptCount < 30:
    return '🟡' (אומדן ראשוני)
  if lastActivityDays > 7:
    return '🟡' (נמדד אבל ישן — לרענן)
  return '✅' (יציב)
```

### בכל מקום שהסימן מופיע

| הקשר | מה הסימן מייצג |
|---|---|
| ליד שם תלמידה (Class View) | מינימום על פני 5 סטרנדים — אם **סטרנד אחד** במצב ⚫, הסימן הכללי 🟡 |
| ליד סטרנד (Student View §3.2.1) | הסטרנד הספציפי הזה |
| ליד משימת ראמ"ה (Student View §3.2.4) | הסטרנד שהמשימה נשענת עליו |
| ליד תא במשבצת (Class View table) | האם הקריאה של המשימה הספציפית הזו אמינה — אם לא, להציג ◌ במקום ערך |

### למה זה חשוב

קולן 2025 + Pashler et al. עוד הזכירו שהצגת מספר בלי ביטחון = "False Confidence". המורה רואה "70%" ומתחילה לפעול עליו — אבל הוא בנוי על 6 ניסיונות. **המערכת חייבת להגיד "אני לא יודעת מספיק" באופן בולט.**

## §8 — Pulse Toggle — Behavior Spec

### מצב יומי (`Daily mode`)

- ברירת מחדל בכל כניסה.
- כל הנתונים = הקריאות העדכניות ביותר.
- הטבלה מציגה את כל 10 העמודות, אבל פעימות עתידיות מוצללות באפור.
- שורת סיכום בתחתית: "מה השתנה היום/השבוע."

### מצב Snapshot

- המורה בוחרת פעימה ספציפית (1/2/3).
- המערכת קוראת `state.events` ומסננת לפי `timestamp` בתוך טווח הפעימה:
  - פעימה 1: 1.9 — 31.10
  - פעימה 2: 1.1 — 28.2
  - פעימה 3: 1.5 — 30.6
- מציגה את הסטטוס **כפי שהיה בסוף הפעימה** (אם הסתיימה) או **כפי שהוא עכשיו** (אם הפעימה פעילה).
- הטבלה מצטמצמת — רק המשימות של הפעימה הנבחרת.
- מאפשרת השוואה: "מי עברה בפעימה 1?" — שאלה שונה מ"איפה הילדה עכשיו".

### החלטה פתוחה לסוכן הקוד

> איך לקרוא את "מצב סוף פעימה 1" אם הפעימה כבר עברה? יש 2 אופציות:
> 1. **Replay** — לקרוא את `state.events` עד `2026-10-31 23:59` ולחשב BKT מחדש. יקר אבל מדויק.
> 2. **Snapshot file** — לכתוב `state.snapshots[pulse_1]` בסוף הפעימה (cron / manual trigger). זול אבל דורש מנגנון.
>
> **המלצה ל-MVP:** אופציה 1 (Replay). פיילוט קטן, חישוב חד-פעמי לכל view. אם זה איטי — נחליף ל-snapshot בפיילוט הבא.
